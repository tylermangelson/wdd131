Unit 1
Variables & Constants
By the end of this unit, you'll understand how programs store and manage information ‚Äî and why naming things well is one of the most important habits you can build as a programmer.
Section 1.1

What Is a Variable?

A variable is a named container that holds a value. Think of it like a labeled box: the label is the name, and whatever's inside is the value. The cool thing about variables is that the contents can change while the label stays the same.

This is how programs remember things. A shopping cart needs to track a running total. A game needs to remember the player's score. A weather app needs the current temperature. Without variables, a program can't hold onto any information ‚Äî it would forget everything the instant it was created.

üß† Mental Model

Think of a whiteboard in a classroom. You can write something on it, erase it, and write something new. The whiteboard is always "the whiteboard," but its contents change. A variable works the same way.

Creating and Using Variables

In JavaScript, you create a variable using the keyword let, followed by a name you choose, an equals sign, and a value.

Basic Variable
let score = 0;
console.log(score); // 0

score = 10; // update the value
console.log(score); // 10

Read this as: "Create a variable named score. Give it the value 0. Later, change that value to 10." The equals sign here is not asking a question ‚Äî it's giving an instruction: "put this value in this box."

Variables Can Hold Different Kinds of Data

Different Data Types
let playerName    = "Amara";      // text (called a "string")
let level          = 5;           // whole number
let healthPercent  = 87.5;        // decimal number
let isAlive        = true;         // true or false (boolean)

Variables Change Over Time

Watching a Variable Change
let temperature = 68;
console.log("Morning temp:", temperature);  // 68

temperature = 74;
console.log("Afternoon temp:", temperature); // 74

temperature = temperature + 3;
console.log("Evening temp:", temperature);   // 77

Notice temperature = temperature + 3 ‚Äî the right side is calculated first using the old value (74), then the result (77) is stored back into the variable. This is a pattern you'll use constantly.

üìù Project: Personal Profile Card

‚ö†Ô∏è Common Pitfalls

Using a variable before creating it ‚Äî JavaScript doesn't know about a variable until the line where you declare it. Using it before that line will crash your program.

Confusing = with == ‚Äî A single = is an instruction: "store this value." It's not asking a question. == is a comparison that asks "are these equal?" We'll cover comparisons in Unit 2.

Variable names are case-sensitive ‚Äî userName and username are two completely different variables in JavaScript.

Vague names cause bugs ‚Äî A variable named x or data tells you nothing. Name variables clearly: currentUserName, orderTotal, isLoggedIn. Use camelCase ‚Äî start lowercase, capitalize each new word.

Section 1.2

Constants ‚Äî Values That Should Never Change

A constant is like a variable that makes a promise: once a value is set, it will never change. In JavaScript, you create constants with const instead of let.

Why would you want something that can't change? Because many values in a program genuinely shouldn't change. Using const makes your intention explicit ‚Äî it tells anyone reading your code: "This value is fixed. Don't touch it."

üß† Mental Model

A const is like a permanent marker versus a regular pencil. A let variable is pencil ‚Äî writable and erasable. A const is permanent marker ‚Äî once written, it stays.

Using const
const SECONDS_IN_A_MINUTE = 60;
const MAX_USERNAME_LENGTH = 20;
const SITE_NAME = "LearnCode";

Many developers write constant names in ALL_CAPS to make it visually obvious they're fixed values. This isn't required, but it's a helpful signal.

What Happens When You Try to Change a Constant
const taxRate = 0.08;
taxRate = 0.09; // TypeError: Assignment to constant variable

JavaScript throws an error ‚Äî and that's actually a good thing. It protects you from accidentally overwriting a value that should stay fixed.

Choosing Between let and const
const MAX_SCORE = 100;     // never changes ‚Üí const
let   currentScore = 0;    // changes as you play ‚Üí let
let   playerName = "Priya"; // could be updated ‚Üí let

currentScore = currentScore + 10;
console.log(`${playerName} scored ${currentScore} out of ${MAX_SCORE}`);

The backtick syntax above (called a template literal) lets you embed variable values directly inside a string using ${}. This is usually cleaner than joining strings with +.

‚ö†Ô∏è Common Pitfalls

You must give const a value immediately ‚Äî const rate; is a syntax error. You have to write const rate = 0.05; on the same line.

Avoid var ‚Äî You'll see it in older code. It works differently from let and const in ways that cause subtle bugs. Modern JavaScript uses let and const.

Section 1.3

Data Types ‚Äî What Kind of Value Is It?

Every value in JavaScript has a type ‚Äî a category that describes what kind of data it is and what you can do with it. Understanding types matters because the same operation can behave very differently depending on what type of data you're working with.

Adding two numbers gives you math. "Adding" two strings joins them together. Adding a number and a string gives you something unexpected ‚Äî and this leads to the concept of type coercion: JavaScript's tendency to automatically convert types in ways that aren't always what you expect.

üß† Mental Model

Types are like units of measurement. "5 meters + 3 meters = 8 meters" makes sense. But "5 meters + 3 kilograms" is nonsense ‚Äî unless you have a system that silently converts one to the other. JavaScript is that system, and it doesn't always choose wisely.

The Basic Data Types
let name        = "Celeste";  // string ‚Äî text in quotes
let age         = 28;         // number ‚Äî no quotes
let score       = 9.4;        // number ‚Äî decimals work too
let isLoggedIn  = false;      // boolean ‚Äî only true or false
let middleName  = null;       // null ‚Äî intentionally empty
let nickname;                 // undefined ‚Äî declared but not assigned

Checking a Value's Type

Using typeof
console.log(typeof "hello");    // "string"
console.log(typeof 42);         // "number"
console.log(typeof true);       // "boolean"
console.log(typeof undefined);  // "undefined"
console.log(typeof null);       // "object" ‚Üê famous JavaScript quirk!

Type Coercion ‚Äî When JavaScript Guesses (Sometimes Wrong)

The Surprising + Operator
let a = "5";  // this is a string, not a number!
let b = 3;

console.log(a + b);  // "53" ‚Äî not 8!
// JavaScript saw a string + number, joined them as text

console.log("10" - 4); // 6 ‚Äî subtraction converts the string to a number
console.log("10" + 4); // "104" ‚Äî addition joins as text
console.log(true + 1);  // 2 ‚Äî true becomes 1
console.log(false + 1); // 1 ‚Äî false becomes 0

Converting Types on Purpose

Explicit Type Conversion
let userInput = "42";              // came from a form as a string
let asNumber  = Number(userInput);  // 42 (now actually a number)
let asString  = String(42);         // "42" (text)
let asBool    = Boolean(0);          // false ‚Äî 0 is "falsy"

‚ö†Ô∏è Common Pitfalls

null vs. undefined ‚Äî Both mean "nothing," but differently. undefined means a variable was never given a value. null means someone deliberately set it to empty. This distinction matters when checking if data exists.

NaN is still type "number" ‚Äî typeof NaN returns "number". Use Number.isNaN() to check for it.

Always use === not == ‚Äî Triple equals checks both value AND type: 5 === "5" is false. Double equals does type coercion first: 5 == "5" is true. Always use === to avoid surprises.

Conditionals
By the end of this unit, you'll understand how programs make decisions ‚Äî how they respond differently based on different inputs, states, and conditions.
Section 2.1

If, Else If, Else ‚Äî Making Decisions

A conditional gives your program the ability to choose. Rather than executing every line in a straight sequence, a conditional lets the program ask a question and take different paths depending on the answer.

This is the foundation of responsive programs ‚Äî programs that behave differently based on what the user does. Without conditionals, every program would do the exact same thing every time, no matter what.

üß† Mental Model

Think of a conditional like a fork in a road. Your program reaches the fork, checks a condition (like reading a road sign), and goes left or right. With else if, you have multiple forks in sequence. With else, you have a final default path if none of the earlier forks applied.

Basic if Statement
let temperature = 85;

if (temperature > 80) {
  console.log("It's hot outside.");
}
// If the condition is true, the block runs. If false, it's skipped.
if-else and if-else if-else
let score = 78;

if (score >= 90) {
  console.log("Grade: A");
} else if (score >= 80) {
  console.log("Grade: B");
} else if (score >= 70) {
  console.log("Grade: C");  // ‚Üê this one runs
} else {
  console.log("Grade: F");
}
// JavaScript checks top to bottom, stops at the first true condition

‚ö†Ô∏è Common Pitfalls

Using = instead of === in a condition ‚Äî if (score = 100) doesn't compare ‚Äî it sets score to 100 and will always be truthy. This is a sneaky bug. Always use === to compare.

Order of else if matters ‚Äî Always check the most specific condition first. If you check score >= 70 before score >= 90, the 90+ case will never run.

Section 2.2

Logical Operators ‚Äî Combining Conditions

Real decisions are rarely based on a single condition. "Should this user see the admin panel?" depends on whether they're logged in and whether they have the right permission. "Should this discount apply?" depends on whether the order is large enough or whether they have a coupon.

Logical operators let you combine multiple conditions: && (AND), || (OR), and ! (NOT).

üß† Mental Model

&& is a security checkpoint with two guards ‚Äî both must wave you through. || is a door with two keys ‚Äî either one opens it. ! flips the answer ‚Äî if something is true, ! makes it false.

AND, OR, NOT
let isLoggedIn = true;
let isAdmin    = false;

// AND: both must be true
if (isLoggedIn && isAdmin) {
  console.log("Access granted.");
} else {
  console.log("Access denied."); // ‚Üê runs (isAdmin is false)
}

// OR: at least one must be true
let hasCoupon   = false;
let orderTotal  = 150;
if (hasCoupon || orderTotal >= 100) {
  console.log("Discount applied!"); // ‚Üê runs (orderTotal qualifies)
}

// NOT: flips the boolean
let userLoggedIn = false;
if (!userLoggedIn) {
  console.log("Please log in.");
}

Combining Operators

Complex Conditions
let age = 17;
let hasParentPermission = true;

if (age >= 18 || (age >= 13 && hasParentPermission)) {
  console.log("Allowed to sign up."); // ‚Üê runs
} else {
  console.log("Not eligible.");
}
// Parentheses control order ‚Äî just like in math

An important behavior called short-circuit evaluation: with &&, if the left side is false, JavaScript never even looks at the right side. This prevents errors when the second check depends on the first being true.

Section 2.3

Switch Statements & the Ternary Operator

When you're matching one variable against a long list of specific values, an if-else if chain can get repetitive. The switch statement is a cleaner alternative. The ternary operator is a compact way to express a simple two-path decision in a single line.

üß† Mental Model

A switch is like a vending machine ‚Äî you enter a code and it routes to one specific slot. A ternary is a one-question survey: "yes or no?" ‚Üí do this or that.

Switch Statement
let dayOfWeek = "Wednesday";

switch (dayOfWeek) {
  case "Monday":
    console.log("Start of the work week.");
    break; // IMPORTANT: stops here
  case "Wednesday":
    console.log("Midweek checkpoint."); // ‚Üê runs
    break;
  case "Friday":
    console.log("Almost the weekend!");
    break;
  default:
    console.log("Regular day.");
}
Ternary Operator
// condition ? valueIfTrue : valueIfFalse
let age    = 20;
let access = age >= 18 ? "Allowed" : "Denied";
console.log(access); // "Allowed"

let label = score >= 60 ? "Pass" : "Fail"; // good use

// Don't nest ternaries ‚Äî it gets unreadable fast!
// Use if-else instead when logic is more complex
SituationBetter Choice
Matching one value against many specific optionsswitch
Range comparisons (>, <, >=)if-else if
Simple true/false assignment in one lineternary
Complex, multi-condition logicif-else if
‚ö†Ô∏è Common Pitfalls

Forgetting break in a switch ‚Äî Without it, execution "falls through" to the next case and runs that too. Always include break unless you're intentionally using fall-through (and if you are, add a comment explaining why).

Switch doesn't work for ranges ‚Äî Switch uses strict equality. You can't do case score >= 90. For ranges, use if-else if.

Unit 3
Functions
By the end of this unit, you'll understand how to write reusable, well-named blocks of code ‚Äî and why this is one of the most important habits in programming.
Section 3.1

Defining and Calling Functions

A function is a named, reusable block of code that performs a specific task. Instead of writing the same logic over and over, you write it once, give it a name, and call it whenever you need it.

A well-named function tells the reader what the code does before they read how it does it. Programs built from small, well-defined functions are easier to read, debug, and update.

üß† Mental Model

A function is like a recipe. The recipe has a name ("Make Pasta"), a list of ingredients (inputs), and a set of steps (the body). Once you've written the recipe, you don't rewrite it ‚Äî you just say "make pasta" whenever you need it.

Defining and Calling
function greetUser() {
  console.log("Welcome! Let's get started.");
}

greetUser(); // call it once
greetUser(); // call it again ‚Äî same result, zero extra code
Functions with Parameters and Return Values
// Parameter: a named placeholder for input
function greetByName(name) {
  console.log(`Welcome, ${name}!`);
}
greetByName("Aaliya"); // Argument: the actual value you pass in

// Return values: giving something back
function calculateTip(billAmount, tipPercent) {
  let tip   = billAmount * (tipPercent / 100);
  let total = billAmount + tip;
  return total; // sends the value back to the caller
}

let dinnerTotal = calculateTip(64.50, 20);
console.log(`Total with tip: $${dinnerTotal}`); // $77.4
‚ö†Ô∏è Common Pitfalls

Forgetting the parentheses when calling ‚Äî greetUser without () just references the function. greetUser() actually runs it.

return ends the function immediately ‚Äî Any code after a return statement is unreachable and will never run.

Forgetting to use the return value ‚Äî let result = addNumbers(5, 3) stores it. addNumbers(5, 3) alone calculates and immediately discards the result.

  1. What's the difference between a parameter and an argument? Create your own example.
  2. Why is return important? What happens if you forget it?
  3. If you copy-paste the same 10 lines of logic in 8 different places and then find a bug, what problem do you have? How would functions have prevented it?
Section 3.2

Scope ‚Äî Where Variables Live

Scope refers to where a variable is visible and accessible in your code. A variable declared inside a function is local to that function ‚Äî it only exists while that function runs. A variable declared at the top level is global ‚Äî accessible everywhere.

üß† Mental Model

Scope is like the difference between a public bulletin board (global) and a private notebook (local). Information in the notebook only exists while that person is working. Once they close it, it's gone. The bulletin board is always there for everyone.

Local vs. Global Scope
function calculateBonus(salary) {
  let bonusRate = 0.10; // local ‚Äî only exists inside this function
  return salary * bonusRate;
}

calculateBonus(50000);
console.log(bonusRate); // ReferenceError ‚Äî doesn't exist out here!

let appName = "BudgetTracker"; // global ‚Äî accessible everywhere

function showHeader() {
  console.log(`App: ${appName}`); // can access the global
}
Better: Functions That Return Instead of Modify Globals
// Risky way: function silently changes a global
let total = 0;
function addItem(price) { total = total + price; }

// Better way: function takes input, returns output, touches no globals
function addItem(currentTotal, price) {
  return currentTotal + price;
}
let total = 0;
total = addItem(total, 10);
total = addItem(total, 25);
console.log(total); // 35
üí¨ Reflection Questions
  1. In your own words, what does it mean for a variable to be "out of scope"?
  2. Why might relying on global variables make a large program hard to debug?
  3. What's the difference between a function that modifies a global vs. one that takes input and returns output? Which is generally better, and why?
Section 3.3

Arrow Functions ‚Äî A Shorter Way to Write Functions

JavaScript has more than one way to write a function. Beyond the standard function declaration, there are arrow functions ‚Äî a more compact syntax that's become the dominant style in modern JavaScript. You'll see them constantly, so it's important to get comfortable reading and writing them.

Three Ways to Write the Same Function
// 1. Function declaration (what you've learned so far)
function square(n) {
  return n * n;
}

// 2. Arrow function (same thing, shorter)
const square = (n) => {
  return n * n;
};

// 3. Compact arrow function (single expression ‚Äî return is implied)
const square = n => n * n;

// All three do exactly the same thing:
console.log(square(5)); // 25
Arrow Function Examples
const add    = (a, b) => a + b;          // multiple params need ( )
const greet  = name => `Hello, ${name}!`;   // single param ‚Äî no ( ) needed
const sayHi  = () => "Hi there!";         // no params ‚Äî () is required
‚ö†Ô∏è Common Pitfalls

Implicit return only works without curly braces ‚Äî n => n * 2 works. n => { n * 2 } returns undefined because you need an explicit return inside curly braces.

Arrow functions and this ‚Äî Arrow functions handle the this keyword differently. This matters when writing object methods (Unit 5). For now, just know the difference exists.

Unit 3 ‚Äî Synthesis

"Don't Repeat Yourself" (DRY) is a core principle in programming. But functions aren't just about avoiding repetition ‚Äî they're about creating a vocabulary for your program. When you name a function calculateTax or validatePassword, you're creating building blocks. How does thinking in functions change the way you design code? What would a program look like that ignored this principle entirely?

Unit 4
Arrays
By the end of this unit, you'll know how to store, access, loop through, and transform lists of data ‚Äî one of the most essential skills in programming.
Section 4.1

Creating and Accessing Arrays

An array is an ordered list of values stored in a single variable. Instead of making separate variables for every item in a list, an array lets you group related values together.

Real programs are full of lists: a shopping cart, search results, a to-do list, a leaderboard. Arrays are how programs represent and work with lists.

üß† Mental Model

An array is like a row of numbered lockers. Each locker has an address (its index), starting from 0. You can look inside any locker by its number, put something new in, or check how many lockers there are.

Creating and Accessing Arrays
let fruits = ["apple", "banana", "cherry"];
let scores = [98, 74, 85, 91];
let empty  = [];  // an empty array

// Accessing by index (STARTS AT 0!)
console.log(fruits[0]);  // "apple"  ‚Üê first element
console.log(fruits[1]);  // "banana"
console.log(fruits[2]);  // "cherry" ‚Üê last element
console.log(fruits[3]);  // undefined ‚Äî doesn't exist!

// Length and last element
console.log(fruits.length);            // 3
console.log(fruits[fruits.length - 1]); // "cherry" ‚Äî last element pattern

// Adding and removing
fruits.push("date");    // add to end
fruits.pop();           // remove from end
fruits.unshift("fig"); // add to beginning
fruits.shift();          // remove from beginning
‚ö†Ô∏è Common Pitfalls

Off-by-one errors ‚Äî The last element is at index length - 1, not length. fruits[fruits.length] is always undefined.

const arrays can still be modified ‚Äî const list = [1, 2, 3] means you can't reassign the variable, but you can still push, pop, or change elements inside it.

üí¨ Reflection Questions
  1. Why does array indexing start at 0 instead of 1?
  2. What does array.length - 1 give you? Why is that useful?

Section 4.2

Iterating Over Arrays ‚Äî Doing Something With Every Item

Having data in an array is only useful if you can work with each item. Iteration means going through each element one by one ‚Äî to display it, process it, filter it, or transform it.

üß† Mental Model

Iteration is like reading every item on a shopping list, one at a time. You don't skip any, and you do something with each one ‚Äî check it off, add it to your cart, compare the price.

Three Ways to Loop Over an Array
let animals = ["cat", "dog", "rabbit"];

// 1. Classic for loop (use when you need the index)
for (let i = 0; i < animals.length; i++) {
  console.log(`Animal ${i + 1}: ${animals[i]}`);
}

// 2. for...of loop (cleaner ‚Äî use when you just need the value)
for (let animal of animals) {
  console.log(animal);
}

// 3. forEach (arrow function per item)
animals.forEach(animal => console.log(animal.toUpperCase()));
map and filter ‚Äî Transforming and Selecting
let prices = [10, 25, 8, 40];

// map: transform each item into a new array
let discounted = prices.map(price => price * 0.9);

Section 5.1

Creating and Accessing Objects

An object is a collection of related data organized as key-value pairs. Where an array stores an ordered list, an object stores named properties ‚Äî like a structured form or a contact card.

üß† Mental Model

An object is like a contact card. Each field (name, phone, email) is a key, and the information in that field is the value. You look up information by asking for the field name, not a number.

Creating and Accessing Objects
let user = {
  name:      "Priya Mehta",
  age:       29,
  email:     "priya@example.com",
  isPremium: true
};

// Dot notation ‚Äî use for known property names
console.log(user.name);      // "Priya Mehta"
console.log(user.email);     // "priya@example.com"

// Bracket notation ‚Äî use when the key is in a variable
let field = "age";
console.log(user[field]);    // 29

// Adding and updating properties
user.city = "Toronto";       // add new property
user.age  = 30;              // update existing
delete user.isPremium;        // remove a property
Arrays of Objects ‚Äî Very Common Pattern
let students = [
  { name: "Aaliya", grade: 88 },
  { name: "Marcus", grade: 74 },

Section 5.2

Methods and this ‚Äî Objects That Do Things

Objects can contain not just data, but also behavior ‚Äî functions stored as properties. A function that belongs to an object is called a method. The this keyword lets a method refer to the object it belongs to, giving it access to the object's own properties.

üß† Mental Model

If a contact object is a business card, a method is a button on that card. When you press "Send Email," the button knows whose card it belongs to and automatically uses that contact's email address. That self-awareness is what this provides.

Methods Using this
let dog = {
  name:  "Biscuit",
  breed: "Labrador",
  describe() {  // shorthand method syntax
    console.log(`${this.name} is a ${this.breed}.`);
  }
};

dog.describe(); // "Biscuit is a Labrador."
// 'this' refers to the dog object itself
A Counter Object
let counter = {
  count: 0,
  increment() { this.count++; },
  reset()     { this.count = 0; },
  getCount()  { return this.count; }
};

counter.increment();
counter.increment();
console.log(counter.getCount()); // 2
counter.reset();
console.log(counter.getCount()); // 0
‚ö†Ô∏è Common Pitfalls

Don't use arrow functions as object methods ‚Äî Arrow functions don't have their own this. Inside an arrow function method, this refers to the outer scope, not the object. Always use regular function syntax for methods that need this.

Section 5.3

Nested Objects and Destructuring

Real-world data is rarely flat. A user object might have a nested address. A product might have nested ratings. Destructuring is a shortcut for pulling values out of objects into named variables.

Nested Objects and Destructuring
let employee = {
  name:    "Jordan Rivera",
  role:    "Engineer",
  contact: { email: "jordan@co.com", phone: "555-0102" },
  skills:  ["JavaScript", "Python", "SQL"]
};

// Accessing nested values
console.log(employee.contact.email); // "jordan@co.com"
console.log(employee.skills[0]);      // "JavaScript"

// Destructuring ‚Äî extract into local variables
let { name, role } = employee;
console.log(name); // "Jordan Rivera"
console.log(role); // "Engineer"

// Nested destructuring
let { contact: { email } } = employee;
console.log(email); // "jordan@co.com"

// Spread: shallow copy of an object
let original = { a: 1, b: 2 };
let copy     = { ...original, c: 3 };
console.log(copy); // { a: 1, b: 2, c: 3 }
‚ö†Ô∏è Common Pitfalls

Drilling into a missing nested object causes a crash ‚Äî If user.address is undefined, trying user.address.city throws a TypeError. Always check that a nested object exists before going deeper.

Spread creates a shallow copy ‚Äî Nested objects inside the copy are still shared by reference. Changing a nested property on the copy will also change it on the original.

Events
By the end of this unit, you'll understand how browser events work ‚Äî how to listen for user actions and respond to them with JavaScript.
Section 6.1

How Events Work in the Browser

Up to now, your programs ran from top to bottom and stopped. Event-driven programming is fundamentally different: your program waits for things to happen ‚Äî a click, a keystroke, a page load ‚Äî and responds. This is how almost every real web app works.

üß† Mental Model

Think of event-driven code like a front desk receptionist. They don't do one thing and go home ‚Äî they wait, respond to whoever walks in, handle that interaction, and then wait again. Events are the "someone walked in" moments.

Adding an Event Listener
// HTML: <button id="myBtn">Click me</button>
//       <p id="message">Nothing yet.</p>

const button  = document.getElementById("myBtn");
const message = document.getElementById("message");

button.addEventListener("click", () => {
  message.textContent = "Button was clicked!";
});

// The event object gives you details about what happened
button.addEventListener("click", (event) => {
  console.log(event.type);    // "click"
  console.log(event.target);  // the element that was clicked
});
EventTriggers when...
"click"User clicks an element
"dblclick"User double-clicks
"mouseover"Mouse enters an element
"keydown"A key is pressed
"keyup"A key is released
"submit"A form is submitted
"input"Input field value changes
"load"Page finishes loading
‚ö†Ô∏è Common Pitfalls

Running JS before the DOM is ready ‚Äî If your script tag is in the <head>, the HTML hasn't loaded yet and getElementById will return null. Fix: put your script tag at the end of the <body>.

Adding the same listener multiple times ‚Äî Each addEventListener call adds a new listener. Calling it inside a loop means the handler fires multiple times per click.

Section 6.2

Keyboard Events and Form Validation

Buttons are just one type of interactive element. Real apps respond to keyboard input, text fields, and form submissions. Form events let your app listen to and validate user input in real time.

Form Events in Action
// Listen to every keystroke in an input
const input = document.getElementById("searchBox");
input.addEventListener("input", (event) => {
  console.log("Current value:", event.target.value);
});

// Prevent form from reloading the page
const form = document.getElementById("signupForm");
form.addEventListener("submit", (event) => {
  event.preventDefault(); // stops the page from reloading!
  const name = document.getElementById("nameInput").value;
  if (name === "") {
    alert("Please enter your name.");
  } else {
    console.log(`Signed up: ${name}`);
  }
});
‚ö†Ô∏è Common Pitfalls

Forgetting event.preventDefault() on form submit ‚Äî Without it, the page reloads and all your JavaScript work disappears. This is almost always the first thing to call in a submit handler.

event.target.value vs. event.key ‚Äî .value is the full current text in the input. .key is the single key that was just pressed. They serve different purposes.

Section 6.3

Event Delegation ‚Äî One Listener for Many Elements

When you have many similar elements that each need to respond to clicks, attaching an individual listener to each one is inefficient. Event delegation is smarter: attach one listener to the parent, and let it handle events from all its children.

This works because of event bubbling: when you click a child element, the click event travels up the DOM tree to all its ancestors.

Event Delegation
// One listener on the parent handles all list items
const list = document.getElementById("taskList");

list.addEventListener("click", (event) => {
  if (event.target.tagName === "LI") {
    event.target.classList.toggle("completed");
  }
});

// Even items added LATER will work ‚Äî no extra listener needed!
const newItem = document.createElement("li");
newItem.textContent = "New Task";
list.appendChild(newItem);

The DOM

Section 7.1

Understanding and Traversing the DOM

The Document Object Model (DOM) is the browser's representation of an HTML page as a tree of objects that JavaScript can read and manipulate. Every element is a node in this tree, and JavaScript can find, modify, create, or delete any of it.

üß† Mental Model

The DOM is like a family tree. document is the ancestor of everything. html has two children: head and body. Every element on the page is a descendant. You can navigate up to parents, down to children, sideways to siblings.

Selecting Elements
// By ID ‚Äî returns one element
const header  = document.getElementById("mainHeader");

// By CSS selector ‚Äî returns first match
const firstBtn = document.querySelector(".btn");

// By CSS selector ‚Äî returns ALL matches (NodeList)
const allBtns  = document.querySelectorAll(".btn");

// Traversing the tree
const list = document.getElementById("myList");
list.parentElement;      // element containing the list
list.children;           // all child elements
list.firstElementChild;  // first child
list.nextElementSibling; // the element after this one
‚ö†Ô∏è Common Pitfalls

innerHTML vs textContent ‚Äî innerHTML renders HTML tags. textContent displays literal text. Setting innerHTML from user input is a security risk (XSS attacks). Use textContent when you don't need to insert HTML.

NodeList is not an array ‚Äî querySelectorAll returns a NodeList, which lacks methods like map and filter. Convert it first: Array.from(nodeList).

Section 7.2

Creating, Modifying, and Removing Elements

Reading the DOM is only half the power ‚Äî the real capability is changing it. JavaScript can create new elements, update existing ones, and remove elements that are no longer needed. This is how single-page apps update without refreshing.

Section 7.3

Building Interactive Interfaces ‚Äî DOM + Events Together

The real power comes from combining the DOM and events. This is the fundamental loop of interactive web development: user interacts ‚Üí event fires ‚Üí DOM updates.

You'll also see state for the first time: keeping track of what the app currently "knows" so the interface accurately reflects that information.

Keeping the DOM in sync with state ‚Äî Always update your state variable first, then update the DOM to reflect it. Never try to read state back out of the DOM ‚Äî the DOM is just the display, not the source of truth.

Section 8.1

Why Sorting Matters ‚Äî Bubble Sort

Sorting is arranging items in a specific order. It's one of the oldest and most studied problems in computer science. This unit goes deeper than calling .sort() ‚Äî you'll understand how sorting works, and how to think about the cost of an algorithm.

üß† Mental Model

Imagine a deck of unsorted cards. One approach: go through the deck and whenever two adjacent cards are in the wrong order, swap them. Keep repeating until no swaps are needed. That's bubble sort ‚Äî not the fastest, but perfectly clear to understand.

Bubble Sort
function bubbleSort(arr) {
  let n = arr.length;

  for (let i = 0; i < n - 1; i++) {
    for (let j = 0; j < n - 1 - i; j++) {
      if (arr[j] > arr[j + 1]) {
        // Swap adjacent elements
        [arr[j], arr[j + 1]] = [arr[j + 1], arr[j]];
      }
    }
  }
  return arr;
}

console.log(bubbleSort([5, 3, 8, 1, 4])); // [1, 3, 4, 5, 8]

For an array of n items, bubble sort compares up to n √ó n pairs in the worst case. If the array doubles in size, the work roughly quadruples. This is called O(n¬≤) time complexity.

Section 8.2

Selection Sort and Insertion Sort

Bubble sort's comparison pattern is quite random ‚Äî it swaps whenever it finds a misplaced pair. Selection sort and insertion sort each represent a more purposeful approach.

Selection Sort ‚Äî Find the Smallest, Place It First
function selectionSort(arr) {
  let n = arr.length;

  for (let i = 0; i < n - 1; i++) {
    let minIndex = i;
    for (let j = i + 1; j < n; j++) {
      if (arr[j] < arr[minIndex]) minIndex = j;
    }
    if (minIndex !== i) {
      [arr[i], arr[minIndex]] = [arr[minIndex], arr[i]];
    }
  }
  return arr;
}
Insertion Sort ‚Äî Build a Sorted Section One Item at a Time
function insertionSort(arr) {
  for (let i = 1; i < arr.length; i++) {
    let current = arr[i];
    let j = i - 1;

    while (j >= 0 && arr[j] > current) {
      arr[j + 1] = arr[j];
      j--;
    }
    arr[j + 1] = current;
  }
  return arr;
}
// Like sorting cards in your hand ‚Äî pick up each card
// and slide it into the right place among the sorted ones

Section 8.3

Merge Sort ‚Äî Divide and Conquer

The three previous algorithms all scale as O(n¬≤) ‚Äî they become unacceptably slow on large datasets. Merge sort is a fundamentally different approach: instead of solving the whole problem at once, break it into smaller subproblems, solve each one, and combine the results. This is called divide and conquer.

üß† Mental Model

Imagine sorting a huge stack of papers. Split the stack in half. Each half further splits until you're holding a single sheet. Then merge: two single sheets are trivially sorted into a pair. Two pairs merge into a sorted group of four. Continue merging. Each merge is simple; the power comes from doing many small, easy merges.

Merge Sort
function merge(left, right) {
  let result = [], i = 0, j = 0;

  while (i < left.length && j < right.length) {
    if (left[i] <= right[j]) {
      result.push(left[i++]);
    } else {
      result.push(right[j++]);
    }
  }
  return result.concat(left.slice(i)).concat(right.slice(j));
}

function mergeSort(arr) {
  if (arr.length <= 1) return arr;  // base case ‚Äî stop splitting

  let mid   = Math.floor(arr.length / 2);
  let left  = mergeSort(arr.slice(0, mid));
  let right = mergeSort(arr.slice(mid));

  return merge(left, right);
}

console.log(mergeSort([5, 3, 8, 1, 4])); // [1, 3, 4, 5, 8]

Appendix C ‚Äî Debugging Strategies

When something breaks, try these steps in order:

1. Read the error message. It tells you the error type and the exact line number. Don't skip this step.

2. Add console.log(). Print the value of variables before and after the problematic line to trace what's happening.

3. Check types with typeof. Many beginner bugs are type mismatches in disguise.

4. Isolate the problem. Comment out code until the bug disappears, then slowly uncomment to narrow down the cause.

5. Use browser DevTools. Press F12. The Console, Sources tab (with breakpoints), and Network tab are your tools.

6. Rubber duck debugging. Explain your code out loud, line by line. You'll often find the bug while explaining it to an imaginary rubber duck.

7. Take a break. Fresh eyes catch what tired eyes miss.
